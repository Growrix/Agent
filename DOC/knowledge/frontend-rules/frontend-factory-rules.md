# Frontend Factory Rules

Canonical execution contract for any frontend generated by this OS when the goal is a professional, production-ready product experience instead of a one-off mockup.

## Purpose

Use this file to keep planning, implementation, review, and audit aligned around one deterministic contract.

This rule set exists to prevent the repeated failure pattern:

- planner leaves design decisions unresolved
- developer fills gaps with generic composition
- tests are scaffolded but not executable
- build compiles yet still needs recurring manual repair

## Planning contract

`frontend_planner` MUST emit an `execution_contract` block inside `frontend.json`.

Minimum required fields:

```json
{
  "execution_contract": {
    "required_scripts": [
      "lint",
      "typecheck",
      "test",
      "test:unit",
      "test:a11y",
      "e2e:smoke",
      "e2e:full",
      "build",
      "audit:frontend",
      "release:check"
    ],
    "mandatory_smoke_journeys": [
      "home_render_no_console_errors",
      "primary_navigation_responsive",
      "theme_switcher_persists",
      "mobile_bottom_nav_routes",
      "auth_modal_mode_switch",
      "primary_conversion_reachable",
      "content_route_resolves",
      "contact_route_resolves"
    ],
    "route_coverage_status": "passed|failed",
    "route_coverage_notes": ["..."],
    "required_release_evidence": [
      "lint_zero_warning",
      "typecheck_zero_error",
      "unit_pass",
      "a11y_smoke_pass",
      "e2e_smoke_pass",
      "build_pass",
      "spec_diff_pass",
      "frontend_self_audit_pass"
    ]
  }
}
```

The planner MUST fail the run if this contract is absent, incomplete, or contradictory.

## Page-brief contract

Per-page briefs MUST provide enough detail for smoke and release validation to be derived without guesswork.

Required planning signals:

- primary and secondary routes
- conversion-critical route and CTA path
- content route(s) that must resolve
- mobile and desktop navigation expectations
- mandatory UX infrastructure on every relevant surface
- dynamic-state expectations when a route is not static

## Developer contract

`frontend_developer` MUST treat the planner's `execution_contract` as binding.

Required implementation behavior:

- generate every required script in `package.json`
- implement real executable smoke tests for every `mandatory_smoke_journeys[]` id
- preserve journey ids verbatim in smoke test titles or comments so audits can trace them
- keep `audit:frontend` as a real executable script that produces evidence
- keep `release:check` as a real executable aggregate gate, not a placeholder alias

Broader non-critical suites may be scaffolded, but mandatory smoke journeys MUST NOT be left as `todo`, `fixme`, or commented-out placeholders.

## Mandatory smoke journeys

These ids are the default minimum for all public-facing frontend builds unless the brief explicitly narrows scope with justification.

| Journey id | Required proof |
|---|---|
| `home_render_no_console_errors` | home route loads without console, hydration, or uncaught runtime errors |
| `primary_navigation_responsive` | primary navigation works on desktop and mobile |
| `theme_switcher_persists` | theme toggles and persists across navigation/reload |
| `mobile_bottom_nav_routes` | mobile bottom nav renders on mobile breakpoints and routes correctly |
| `auth_modal_mode_switch` | auth modal opens, switches sign-in/sign-up mode, closes, and fallback routes resolve |
| `primary_conversion_reachable` | main conversion surface is reachable from home with the planned CTA path |
| `content_route_resolves` | at least one content-bearing route resolves successfully |
| `contact_route_resolves` | contact or support route resolves successfully |

If the project has no auth, content, or contact surface, the planner must replace that journey with a project-specific equivalent and record the justification.

## Release gate order

`release:check` MUST run in this order:

1. runtime-root detection
2. dependency validation
3. `lint`
4. `typecheck`
5. `test:unit`
6. `test:a11y`
7. `e2e:smoke`
8. `build`
9. `audit:frontend`
10. `e2e:full` for production-candidate builds unless explicitly skipped by policy

The gate MUST fail fast.

## Reviewer and audit expectations

`reviewer` and `system_architect` MUST treat these as blocking failures:

- planner emits no `execution_contract`
- generated app lacks one or more required scripts
- mandatory smoke journeys are missing or TODO-only
- reviewer coverage is stale relative to current frontend constraints
- self-audit omits Q or CC constraints
- release evidence is claimed without executable proof

## Evidence expectations

Production-ready means evidence exists for all of the following:

- command surface present
- mandatory smoke journeys implemented
- frontend self-audit exists with cited paths
- planner-vs-output spec diff passes
- no unresolved route-coverage gaps
- no unresolved critical open questions blocking build fidelity

## Failure codes

- `FRONTEND_FACTORY_EXECUTION_CONTRACT_MISSING`
- `FRONTEND_FACTORY_SMOKE_JOURNEY_MISSING`
- `FRONTEND_FACTORY_RELEASE_GATE_MISSING`
- `FRONTEND_FACTORY_ROUTE_COVERAGE_FAILED`
- `FRONTEND_FACTORY_EVIDENCE_INCOMPLETE`
