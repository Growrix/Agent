# Factory Frontend Developer Agent — Architecture & Build Specification

Version: 1.0
Purpose: Build a new frontend execution agent specifically designed for the Factory architecture.

IMPORTANT:
This agent DOES NOT replace the existing frontend_developer yet.

The existing frontend_developer remains:

* stable
* proven
* production-usable
* backward-compatible

This new agent is:

* experimental
* factory-native
* retrieval-oriented
* scoped-context-driven
* composition-aware

The goal is to evolve frontend execution safely without breaking the existing system.

---

# 1. CORE DECISION

DO NOT modify the current frontend_developer.

Instead:

Create:

```txt
frontend_factory_developer
```

This agent should operate independently.

This allows:

* parallel testing
* benchmarking
* architecture comparison
* safe migration
* rollback capability

This is the correct engineering decision.

---

# 2. WHY THE EXISTING FRONTEND_DEVELOPER SHOULD STAY

The current frontend_developer already:

* understands DOC contracts
* follows planning artifacts
* generates technically valid systems
* works with existing architecture
* already solves many engineering problems

It should not be destroyed prematurely.

The problem is NOT that it is bad.

The problem is:

It was designed before the Factory architecture existed.

Meaning:

It operates like:

```txt
Universal Frontend Generator
```

But the Factory requires:

```txt
Scoped Frontend Executor
```

That is a fundamental architectural difference.

---

# 3. PRIMARY GOAL OF THE NEW AGENT

The new Factory Frontend Developer Agent must:

* consume orchestrated decisions
* consume scoped retrieval context
* implement controlled frontend systems
* avoid global repo reasoning
* avoid uncontrolled design decisions
* avoid random component invention
* avoid architecture drift

This agent should NOT behave like a creative AI.

It should behave like:

# A deterministic implementation engine.

---

OLD MODEL:


NEW MODEL:

```txt
Planner
```

This is critical.

# 5. REQUIRED INPUTS

The new agent should ONLY receive:

## A. FRONTEND CONTRACT

Generated by:

```txt
frontend_planner
```

Contains:

* routes
* pages
* features
* rendering strategy
* state requirements
* SEO requirements
* frontend architecture rules

---

## B. EXPERIENCE CONTRACT

Generated by:

```txt
experience_orchestrator
```

Contains:

* visual style
* section selection
* layout structure
* motion level
* UX density
* theme system
* composition decisions

---

## C. RETRIEVED CONTEXT PACKETS

Generated by:

```txt
context_retriever
```

Contains ONLY:

* relevant components
* relevant sections
* required tokens
* relevant motion presets
* exact dependencies
* compatible layouts

The agent must NEVER retrieve additional unrelated context itself.
